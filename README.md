# GREEN-API Test Framework

QA-фреймворк для автоматизированного тестирования WhatsApp-шлюза [GREEN-API](https://green-api.com).
Реализован как демонстрационное тестовое задание на **Jest + TypeScript**.

---

## Стек технологий

| Инструмент | Назначение |
|-----------|-----------|
| Node.js 18+ | Среда выполнения |
| TypeScript 5 | Типизация на уровне всего проекта |
| Jest 29 | Тест-раннер и утверждения |
| ts-jest | TypeScript-транспилятор для Jest |
| axios | HTTP-клиент |
| dotenv | Загрузка переменных окружения |

## Настройка

```bash
# 2. Скопировать файл с примером
cp .env.example .env
```

Открой `.env` и заполни свои данные:

```dotenv
GREEN_API_BASE_URL=https://api.green-api.com

# Из личного кабинета GREEN-API
GREEN_API_ID_INSTANCE=1234567890
GREEN_API_TOKEN_INSTANCE=abcdef1234567890abcdef1234567890

# Номер WhatsApp — должен совпадать с номером, привязанным к инстансу
# Формат: <код_страны><номер>@c.us
GREEN_API_CHAT_ID=79001234567@c.us

# Используется в негативных тестах — должен быть синтаксически некорректным
GREEN_API_INVALID_CHAT_ID=invalid-chat-id

# Количество сообщений на один запрос getChatHistory
GREEN_API_HISTORY_COUNT=20

# Настройки поллинга для e2e-теста
GREEN_API_POLL_INTERVAL_MS=3000
GREEN_API_POLL_TIMEOUT_MS=60000
```

> **Важно:** перед запуском тестов инстанс GREEN-API должен находиться в состоянии `authorized`.
> Зайди в личный кабинет, открой страницу инстанса и отсканируй QR-код, если статус отличается от «Authorized».

## Покрытые тест-кейсы

### `getStateInstance.spec.ts`
- HTTP 200 при успешном запросе
- Ответ содержит поле `stateInstance`
- `stateInstance` — непустая строка
- `stateInstance` равен `"authorized"` (с диагностической подсказкой при отличии)
- Диагностический лог текущего состояния

### `sendMessage.spec.ts`
**Позитивные:**
- Валидные chatId + message → HTTP 200
- Ответ содержит непустой `idMessage`
- `idMessage` truthy (сообщение принято в очередь)

**Негативные:**
- Отсутствует `chatId` → ошибка (не 200)
- Отсутствует `message` → ошибка (не 200)
- Пустое тело `{}` → ошибка (не 200)
- Тело `null` → ошибка (не 200)
- Некорректный формат `chatId` → ошибка (не 200)
- Сообщение > 4096 символов → ошибка (не 200) *(см. раздел «Известные особенности API»)*

### `getChatHistory.spec.ts`
**Позитивные:**
- Валидный chatId → HTTP 200
- Ответ — массив
- Каждое сообщение содержит обязательные поля (`type`, `idMessage`, `timestamp`, `typeMessage`, `chatId`)
- `timestamp` — положительное число
- `type` равен `"incoming"` или `"outgoing"`
- `count=1` возвращает ≤ 1 сообщения
- `count=5` возвращает ≤ 5 сообщений
- Значение `count` из конфига соблюдается

**Негативные:**
- Отсутствует `chatId` → ошибка (не 200)
- Некорректный `chatId` → ошибка (не 200)
- Пустое тело → ошибка (не 200)
- Тело `null` → ошибка (не 200)
- `count` как строка → 200 или 400 (мягкая проверка)

### `e2e-message-flow.spec.ts`
1. Инстанс в состоянии `authorized`
2. Генерируется уникальное сообщение
3. `sendMessage` возвращает 200 + валидный `idMessage`
4. Поллинг `getChatHistory` находит сообщение в пределах таймаута
5. Проверки найденного сообщения:
   - `textMessage` совпадает с отправленным текстом
   - `chatId` совпадает с конфигом
   - `typeMessage === "textMessage"`
   - `type === "outgoing"`
   - `idMessage` совпадает с полученным от `sendMessage`

## Почему поллинг в e2e-тесте?

Доставка сообщений в WhatsApp **асинхронна**. После того как `sendMessage` возвращает HTTP 200
(сообщение принято GREEN-API), может пройти несколько секунд, прежде чем оно появится в `getChatHistory`.

Фиксированный `sleep()` был бы либо слишком коротким (нестабильно), либо избыточно длинным.
Хелпер `pollUntil` проверяет историю каждые `GREEN_API_POLL_INTERVAL_MS` миллисекунд и останавливается,
как только сообщение найдено — тест получается одновременно **быстрым** и **надёжным**.

## Ограничения интеграционных тестов

| Ограничение | Влияние |
|-------------|---------|
| Тесты обращаются к реальному GREEN-API — нужны валидные учётные данные | Нельзя запускать в CI без секретов |
| Доставка сообщений зависит от задержек сети WhatsApp | Таймаут e2e-теста должен быть с запасом (≥ 60 с) |
| `GREEN_API_CHAT_ID` должен совпадать с номером инстанса | Требуется самоотправка; разные номера — два инстанса |
| Rate-лимиты GREEN-API публично не задокументированы | Параллельный запуск сьютов может вызвать ошибки 429 — установлен `maxWorkers: 1` |
| Поведение негативных тестов может отличаться от документации | См. раздел «Известные особенности API» |

## Известные особенности API

1. **Некорректный chatId в sendMessage / getChatHistory**
   По спецификации ожидается HTTP 400. GREEN-API может вернуть 400 или другой не-2xx код
   в зависимости от слоя валидации. Тесты проверяют `status !== 200`, а не `status === 400`,
   чтобы честно отражать реальное поведение интеграции.

2. **Очень длинные сообщения**
   GREEN-API публично не документирует максимальную длину сообщения. Тест для сообщений
   в 5000 символов проверяет `status !== 200`. Если API принимает длинные сообщения и возвращает 200 —
   измени утверждение на `toBe(200)` и добавь заметку в этот раздел.

3. **getChatHistory с count как строкой**
   API может привести значение к числу или вернуть 400. Тест принимает оба варианта.

4. **Пустая история чата**
   Если в целевом чате нет сообщений, проверки полей в getChatHistory пропускаются
   (массив пуст, проверять нечего). Это корректное поведение, а не пробел в тестировании.

5. **idMessage в e2e**
   Ожидается, что `idMessage` из `sendMessage` совпадёт с `idMessage` в истории.
   Если GREEN-API присваивает другой ID после доставки — утверждение упадёт.
   В таком случае скорректируй или удали эту проверку.

## Архитектурные решения

- **`validateStatus: () => true`** — axios настроен не бросать исключения на не-2xx ответы,
  поэтому тесты могут проверять ответы с ошибками напрямую без try/catch.
- **`maxWorkers: 1`** — предотвращает конкурентные запросы, которые могут вызвать rate-лимиты
  или состояние гонки на одном инстансе.
- **Гарды в `beforeAll`** — каждый сьют проверяет `stateInstance === "authorized"` перед запуском,
  выдавая понятное сообщение об ошибке вместо неочевидных сбоев дальше по цепочке.
- **Raw-перегрузки (`sendMessageRaw`, `getChatHistoryRaw`)** — позволяют негативным тестам отправлять
  произвольные payload без борьбы с системой типов TypeScript.
- **Централизованный построитель URL** — добавление нового метода GREEN-API требует одной строки
  в `greenApiClient.ts`, без дублирования URL.
